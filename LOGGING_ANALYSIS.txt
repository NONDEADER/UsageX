# COMPREHENSIVE LOGGING ARCHITECTURE ANALYSIS
# Claude Usage Tracker Extension v5.2.7 (Firefox)
# Analysis Date: 2026-06-01

## EXECUTIVE SUMMARY

The Claude Usage Tracker extension implements a sophisticated debug logging system with:
- Centralized log storage using browser.storage.local API
- Conditional debug mode with time-based expiration
- Real-time log viewer UI with auto-refresh capabilities
- Multi-source logging (background, content scripts)
- Log level categorization (debug, warn, error)
- Circular buffer with 1000 log entry limit
- Formatted timestamps with millisecond precision
- Color-coded UI display by log level and source

---

## 1. LOG STORAGE & DATA STRUCTURE

### Storage Mechanism
- **Storage API**: rowser.storage.local (IndexedDB-backed in Firefox)
- **Storage Key**: debug_logs
- **Format**: Array of log entry objects

### Log Entry Data Structure
Each log entry object contains:
`javascript
{
  timestamp: string,    // "HH:MM:SS.mmm" (24-hour format with milliseconds)
  sender: string,       // "background" or "content:page-title"
  level: string,        // "debug", "warn", or "error"
  message: string       // Processed and stringified message
}
`

### Timestamp Format Implementation
- Format: ISO-like with 24-hour time and fractional seconds
- Precision: 3 decimal places (milliseconds)
- Generated using: 	oLocaleString('default', {...fractionalSecondDigits: 3})
- Example: "14:35:42.123"

### Storage Limits
- **Maximum Entries**: 1000 logs (circular buffer)
- **Eviction Policy**: FIFO (First In First Out) - oldest entries removed
- **Implementation**: if (logs.length > 1000) logs.shift();

## 2. LOGGING IMPLEMENTATION

### RawLog Function (bg-components/utils.js)
Core logging function used across the extension.

**Signature**:
`javascript
async function RawLog(sender, ...args)
`

**Parameters**:
- sender: String identifying the source ("background" or "content:...")
- ...args: Variable arguments to log
- Optional first argument: Log level ("debug", "warn", "error")

**Processing Steps**:
1. Parse log level from first argument (if valid level string)
2. Check debug mode status from storage
3. Check timestamp for debug mode expiration
4. Log to console with [UsageTracker] prefix
5. Create timestamp with millisecond precision
6. Serialize all arguments:
   - Errors: Convert to stack traces
   - Objects: JSON.stringify with error handling
   - Primitives: String conversion
7. Create log entry object
8. Retrieve current logs from storage
9. Append new entry
10. Enforce 1000-entry limit
11. Save back to storage

### Debug Mode Control
- **Storage Key**: debug_mode_until
- **Value Type**: Unix timestamp (milliseconds)
- **Comparison**: Current timestamp vs stored expiration
- **Permanent Mode**: Uses sentinel value 8640000000000000 (max safe int)
- **Check Logic**: (!debugUntil || debugUntil <= now) && !FORCE_DEBUG

### Log Level Support
- **debug**: Default level, logged to console.log()
- **warn**: Logged to console.warn(), background styling applied
- **error**: Logged to console.error(), error styling applied
- **Usage**: wait Log("warn", "Warning message")

### Content Script Logging (content-components/content_utils.js)
Nearly identical implementation with:
- Sender format: "content:"
- Same log entry structure and storage mechanism
- Global FORCE_DEBUG flag from storage

## 3. DEBUG MODE MANAGEMENT

### Debug Mode States
1. **Disabled**: No logs unless FORCE_DEBUG is true
2. **Temporary**: Enabled for 1 hour via button
3. **Permanent**: Indefinite until manually disabled

### Debug Mode Duration
- **Default**: 1 hour (60 * 60 * 1000 milliseconds)
- **Calculated**: 
ew Date(Date.now() + 60 * 60 * 1000).getTime()

### Activation Methods
1. **Normal Mode**: Button "Enable Debug Mode (1 hour)"
   - Sets expiration to 1 hour from now
   - Display updates with countdown timer

2. **Permanent Mode**: Button "Enable Debug Mode (Permanent)"
   - Sets debug_mode_until to sentinel value
   - Never expires until user disables

### Debug Status Display
In debug.js:
`javascript
function updateDebugStatus() {
  const isPermanent = debugUntil === PERMANENT_DEBUG;
  const timeLeft = isEnabled && !isPermanent ? Math.ceil((debugUntil - now) / 60000) : 0;
  
  // Displays: "Debug mode enabled (X minutes remaining)" or similar
}
`

        const logLine = document.createElement('div');
        logLine.className = 'log-line';
        logLine.dataset.level = log.level || 'debug';
        
        // Create and append timestamp, sender, message spans

    });

`

#### clearLogs()
`javascript
function clearLogs() {
  browser.storage.local.set({ debug_logs: [] })
    .then(showLogs);  // Refresh display immediately
}
`

#### Auto-Refresh System
- **Interval**: 5 seconds (5000 ms)
- **Checkbox Control**: #autoUpdate checkbox gates refresh
- **Functions**:
  `javascript
  function startAutoRefresh() {
    if (!autoRefreshInterval) {
      autoRefreshInterval = setInterval(() => {
        if (document.getElementById('autoUpdate').checked) {
          showLogs();
        }
        updateDebugStatus();  // Update countdown too
      }, 5000);
    }
  }
  
  function stopAutoRefresh() {
    if (autoRefreshInterval) {
      clearInterval(autoRefreshInterval);
      autoRefreshInterval = null;
    }
  }
  `

#### Scroll Behavior
`javascript
function scrollToBottom() {
  const preElement = document.getElementById('logs');
  preElement.scrollTop = preElement.scrollHeight;  // Auto-scroll to bottom
}
`

### Initialization
- On page load: showLogs(), updateDebugStatus(), startAutoRefresh()
- Cleanup on unload: Auto-refreshing stopped via eforeunload event
- Auto-refresh stops when debug mode expires

## 6. LOG CLEARING & MANAGEMENT

### Clear Mechanism
- **Function**: clearLogs() in debug.js
- **Process**:
  1. Set debug_logs storage key to empty array []
  2. Call showLogs() to immediately refresh UI
  3. Display becomes empty

### Automatic Eviction
- **Policy**: Circular buffer with 1000 entry limit
- **Eviction Method**: logs.shift() removes oldest entry
- **Timing**: Checked on every new log entry

### Manual Management
- Clear button available in debug UI
- Force clear available programmatically
- No retention policy beyond 1000 entries

### Storage Persistence
- Logs persist across extension reloads
- Logs clear only via:
  - User clicking Clear button
  - Extension uninstall (clears all storage)
  - Programmatic clearing from background script

## 8. MULTI-SOURCE LOGGING ARCHITECTURE

### Log Sources

#### Background Script (background.js)
- Uses: wait Log(...args) wrapper around RawLog
- Sender: "background"
- Logs: API calls, state changes, alarm triggers
- Example: wait Log("Processing response...");

#### Content Script (content_utils.js)
- Uses: sync function Log(...args) (independent implementation)
- Sender: "content:"
- Logs: DOM interactions, UI updates
- Example: wait Log("warn", "Element not found");

#### Error Handlers (content_utils.js)
`javascript
// Global error handler
window.addEventListener('error', async (event) => {
  await logError(event.error);
});

// Unhandled promise rejection
window.addEventListener('unhandledrejection', async (event) => {
  await logError(event.reason);
});
`

### Log Processing Pipeline
1. RawLog(sender, ...args) or Log(...args) called
2. Level parsing from first arg
3. Debug mode check
4. Console output (for developer debugging)
5. Timestamp generation with milliseconds
6. Argument serialization (errors to stacks, objects to JSON)
7. Log entry creation
8. Storage retrieval
9. Entry appending
10. Limit enforcement (1000 max)
11. Storage saving

## 10. KEY ARCHITECTURAL DECISIONS

### 1. Centralized Storage
- Single debug_logs array in browser.storage.local
- Pros: Simple implementation, atomic updates, persistent
- Cons: Entire array loaded/saved on each log

### 2. Circular Buffer with Limit
- 1000 entry maximum
- Pros: Prevents unbounded storage growth
- Cons: Oldest logs lost without export

### 3. Conditional Logging
- Debug mode toggle with expiration
- Pros: Can disable in production, flexible durations
- Cons: Timestamp-based comparison overhead

### 4. Per-Entry Timestamp
- Millisecond precision, formatted string
- Pros: Human readable in UI, precise timing
- Cons: Stored as string (less queryable), formatting overhead

### 5. Source-Based Styling
- Color coding by component (background vs content)
- Pros: Visual separation of concerns
- Cons: Limited to two sources in current UI

### 6. Real-time Auto-Refresh
- 5-second polling interval
- Pros: Live log view, checkbox toggle control
- Cons: Inefficient (full re-render), no event-based updates

## 11. ADVANCED FEATURES

### Serialization of Complex Objects
`javascript
message: args.map(arg => {
  if (arg instanceof Error) {
    return arg.stack || ${arg.name}: ;
  }
  if (typeof arg === 'object') {
    if (arg === null) return 'null';
    try {
      return JSON.stringify(arg, Object.getOwnPropertyNames(arg), 2);
    } catch (e) {
      return String(arg);
    }
  }
  return String(arg);
}).join(' ')
`

Features:
- Error stack traces preserved
- Objects prettified with 2-space indentation
- Safe fallback if stringification fails
- All arguments joined with spaces

### Error Handler Integration
`javascript
async function logError(error) {
  if (!(error instanceof Error)) {
    await Log("error", JSON.stringify(error));
    return;
  }
  await Log("error", error.toString());
  if ("captureStackTrace" in Error) {
    Error.captureStackTrace(error, logError);
  }
  await Log("error", JSON.stringify(error.stack));
}
`

### Manifest Configuration
- debug.html included in web_accessible_resources
- No special permissions needed for debug page
- Works in both Firefox and Chrome
